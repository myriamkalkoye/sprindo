# Definition of Ready (DoR) — Projet Sprindo

Une user story peut entrer dans un sprint si les conditions suivantes sont respectées :

## 1. Formulation claire
-Les User Story sont rédigées dans Jira au format standard :
En tant que [rôle], je veux [fonctionnalité], afin de [bénéfice].
-Chaque User Story est claire, compréhensible et cohérente avec le backlog du projet.
- La description fonctionnelle est compréhensible par toute l’équipe.

## 2. Critères d’acceptation
- Les critères d’acceptation sont définis, clairs et validés par le PO.
- Les cas particuliers sont identifiés.
## 3. Spécifications connues
- Les champs nécessaires à l’implémentation sont spécifiés.
    - Projet : id, nom, description
    - Sprint : id, numéro, objectif, début, fin
    - Tâche : id, titre, description, statut, assignéA
- Le comportement attendu est documenté pour les cas normaux et d’erreur.

## 4. Priorisation et estimation
- La user story est priorisée dans le backlog par le PO.
- L’équipe a estimé la complexité (ex. story points).
- Les User Story sont estimées en points (ou charge de travail).

## 5. Dépendances identifiées
- Les dépendances techniques ou fonctionnelles sont connues 
- Les dépendances avec d’autres stories sont identifiées et ne bloquent pas le développement. 
- Les impacts sur les autres entités (projets, sprints, utilisateurs) sont clarifiés.

## 6. Validation préalable
- La user story a été revue lors du backlog refinement.
- L’équipe a revisité les User Story une par une et est d’accord pour les réaliser.
- Chaque User Story a été validée par le Product Owner (PO).
